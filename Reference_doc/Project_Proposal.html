Project Proposal

Keystone RAIN Platform Deployment for Novartis MMF Use Cases

Version: 2.0 | Date: Current Session | Status: Enhanced Proposal with Knowledge Base Service

Executive Summary

Keystone proposes deploying its RAIN Data Platform to address three critical Managed Markets Finance (MMF) use cases for Novartis, delivering significant time savings, improved forecast accuracy, and automated workflow capabilities.

Key Value Propositions

  • Chargeback Process Automation: Reduce processing time from 25 minutes to <15 minutes per kickout, automating 99.9% of "Customer not found" resolutions
  • Medicaid GTN Forecasting: Improve forecast accuracy to achieve 1% variance in out-of-period adjustment by leveraging 867 sellout data
  • Medicaid Rebate Automation: Reduce invoice review time from 60 minutes to <10 minutes for sub-500K claims through automated error detection

Platform Capabilities

  • Knowledge Base Service: Automated rule book management (GPO Rosters, DDD Matrix, Eligibility Guides)
  • LLM-Powered Extraction: 98%+ accuracy in structured data extraction from unstructured documents
  • Workflow Automation: 8-step chargeback process automation with conditional logic
  • External System Integration: Seamless integration with SAP 4 HANA, Model N, and federal databases
  • RAIN Data Encoding: Unified feature encoding across all use cases for downstream analytics

1. Engagement Overview

1.1 Business Challenge

Novartis's commercial organization seeks to transform how it leverages data to power best-in-class AI models, improving efficiency and driving meaningful increases in operating profitability. The challenge lies in preparing finest-grain data (claim-level dispense data, EDI terms) in the right format for AI models in commercial pharmaceutical settings.

1.2 Solution Approach

Keystone will deploy its RAIN Data Platform on Novartis's chosen cloud environment, providing:

  • Unified document management and multi-source data ingestion
  • LLM-powered structured data extraction
  • Knowledge Base Service for automated rule application
  • Workflow automation with extensible processor framework
  • External system integration (SAP, Model N, federal databases)
  • RAIN data encoding for downstream analytics and forecasting

1.3 Engagement Structure

  • Phase 1A (16-18 weeks): MVP workflow automation and models deployed on Novartis environment
  • Phase 1B (8-12 weeks): Productionization, full system integrations, and model improvements
  • Total Engagement: 24-30 weeks (6-7.5 months)

2. Three MMF Use Cases

Use Case 1: Chargeback Process Automation

Problem Statement

  • 50-200 chargebacks "kicked-out" of Model N daily due to "Customer not found" errors
  • Each kickout requires ~25 minutes of manual processing by 5 full-time analysts
  • 8 highly manual and repetitive steps per kickout
  • Must be resolved within 5 days

Solution

Deploy automated workflow using:

  • Process Automation: 8-step automated workflow with conditional logic
  • LLM-Based Term Extraction: Extract customer identifiers and contract specifications
  • Knowledge Base Service: Query GPO Rosters, DDD Matrix, and Eligibility Guides
  • External API Calls: Automated queries to SAP 4 HANA and federal databases (DEA, HIBCC, HRSA)
  • Class-of-Trade Determination: Automated COT determination using DDD Matrix rules
  • SAP Customer Creation: Programmatic customer creation in SAP 4 HANA

8-Step Workflow

  1. Extract Identifiers (HIN, DEA, Address) from chargeback EDI
  2. SAP Customer Existence Check
  3. Query Knowledge Base for Contract Eligibility
  4. GPO Roster Validation
  5. Federal Database Lookup (DEA/HIBCC/HRSA)
  6. Class-of-Trade Determination (using DDD Matrix)
  7. SAP Customer Creation Request Generation
  8. Chargeback Resolution & Compliance Documentation

Success Criteria

  • Time Reduction: 10+ minute reduction per kickout (target: <15 minutes total)
  • Automation Rate: 99.9% automated resolution rate
  • Reusability: Demonstrate reusable platform components extensible to additional kickout types

Use Case 2: Medicaid GTN Forecasting

Problem Statement

  • Claim-level dispensing data has 3+ month reporting lag
  • Creates reporting gap that reduces ability to create accurate gross-to-net forecasts
  • At time of forecast, missing 3+ recent months of utilization data
  • Central finance team needs to forecast Medicaid rebates through end-of-year with incomplete data

Solution

Deploy forecasting model that:

  • Leverages Leading Indicators: Uses 867 sellout data with shorter reporting lag
  • Models Lead/Lag Relationships: Analyzes relationship between 867 sellout and Medicaid utilization
  • Provider ID Matching: Reconciles provider ID mismatches between 867 and Medicaid data
  • Multi-Granularity Analysis: Analyzes at provider, state, and national levels
  • 4-Month Prediction Horizon: Predicts utilization 4 months ahead of reported data

Model Specifications

  • Forecast Target: Medicaid utilization (units dispensed) per brand at national level
  • Brand Coverage: All active Novartis brands with 2+ years utilization history
  • Prediction Horizon: ~4 months ahead of reported utilization
  • Prediction Unit: Week or month (determined during discovery)

Success Criteria

  • Forecast Accuracy: Improve GTN forecast accuracy to achieve 1% variance in out-of-period adjustment
  • Reusability: Reusable forecasting method easily executable on receipt of new data
  • Impact Analysis: Demonstrate improvement in central finance team's ability to hit KPIs

Use Case 3: Medicaid Rebate Invoice Automation

Problem Statement

  • Medicaid rebate processing team (5 FTE) overburdened reviewing ~5,000 rebate invoices annually
  • Each review takes ~45 minutes to several days
  • ~80% of claims are sub-500K but represent only 11% of rebates paid
  • No ability to triage/prioritize problematic vs. non-problematic claims
  • Novartis disputes ~$50M in rebates annually (~5% of total)

Solution

Deploy multi-stage AI model that:

  • LLM-Powered Term Extraction: Extract key terms from unstructured invoices (date, claimant, NDC, UOM, etc.)
  • Probabilistic Error Detection: Multi-stage model to identify errors (UOM, duplicates within invoice, duplicates across invoices)
  • Invoice Triage: Flag invoices above probability threshold for human review, approve below threshold
  • Focus on Sub-500K Claims: Initial model focuses on automating checks for sub-500K claims
  • Dispute Prediction: Assess likelihood of successful dispute and time to resolution

Error Detection Types

  • UOM Errors: Units of measure validation
  • Duplicates within Invoice: NDC + date matching
  • Duplicates across Invoices: Historical invoice matching
  • 340B Duplicates: Medicaid exclusion file check (Phase 1B)

Success Criteria

  • Time Reduction: Substantial reduction in time-to-review for sub-500K invoices (target: <10 minutes)
  • Accuracy: High-accuracy error detection (validated via backtesting)
  • Productization: Demonstrate potential for expansion to additional error types and larger claims

6. Risk Mitigation

6.1 Timeline Risks

Risk Impact Mitigation
System access delays High - Blocks project start Early access requests (Week 1), parallel workstreams
API integration complexity Medium - Delays MVP deployment Early API exploration, sandbox testing, fallback to file-based
Data quality issues Medium - Delays model development Data quality assessment in Weeks 1-2, parallel data cleaning
Cross-workstream dependencies Medium - Blocks multiple workstreams Prioritize shared components (KB service, RAIN platform) in Weeks 1-4

6.2 Technical Risks

  • Federal Database Data Lags: Implement caching and retry logic with manual override
  • API Rate Limits: Implement throttling and queue-based processing
  • Model Accuracy: Extensive backtesting and validation before deployment
  • Integration Failures: Graceful degradation and exception handling
