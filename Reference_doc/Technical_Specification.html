Technical Specification

Keystone RAIN Platform for Novartis MMF Use Cases

Version: 2.0 | Date: Current Session | Status: Enhanced Specification with Knowledge Base Service

1. System Overview

The Keystone RAIN Platform is a comprehensive document management and workflow automation system designed to support three primary Managed Markets Finance (MMF) use cases for Novartis:

  • Chargeback Process Automation - Automated resolution of "Customer not found" chargeback kickouts
  • Medicaid GTN Forecasting - Improved accuracy of Medicaid gross-to-net forecasts using 867 sellout data
  • Medicaid Rebate Invoice Automation - Automated triage and error detection for rebate invoices

2. Architecture Components

2.1 Knowledge Base Service NEW

2.1.1 Overview

The Knowledge Base Service manages structured rule books, reference documents, and eligibility matrices that drive automated workflow decisions.

2.1.2 Rule Book Types

Rule Book Type Purpose Update Frequency Storage Format
GPO Roster Customer contract eligibility information On contract/amendment receipt Excel, CSV, Google Sheets
DDD Matrix Class-of-trade (COT) determination rules On contract changes Excel, Structured Data
ICCR Eligibility Guide Eligibility verification rules and procedures On contract amendments PDF, Word, Google Docs
Contract Eligibility Specifications Detailed contract terms and eligibility requirements Real-time from contract system API, Database, Structured Data

2.1.3 Technical Specifications

Storage:
  • Rule books stored as DocEX documents in dedicated baskets (gpo_rosters, ddd_matrices, eligibility_guides)
  • Structured rule data indexed in PostgreSQL with full-text search capabilities
  • Vector embeddings stored for semantic search (FAISS or Pinecone)
Ingestion:
  • LLM-powered extraction of structured data from rule books
  • Automatic versioning and change tracking
  • Support for Google Drive API integration for automated updates
Query Interface:
  • Fast lookup by indexed fields (customer ID, contract number, etc.)
  • Semantic search using RAG services for complex queries
  • REST API endpoints for programmatic access
Version Control:
  • Track all changes to rule books over time
  • Maintain audit trail of rule modifications
  • Support rollback to previous rule versions

2.2 Enhanced Chargeback Workflow ENHANCED

2.2.1 8-Step Workflow Processors

Step Processor Input Output Integration
1 IdentifierExtractionProcessor Chargeback EDI HIN, DEA, Address LLM Adapter
2 SAPCustomerCheckProcessor Customer identifiers Customer exists (Y/N) SAP 4 HANA API
3 ContractEligibilityProcessor Contract specifications Eligibility validation Knowledge Base Service
4 GPORosterValidationProcessor Customer ID, Contract Roster match result Knowledge Base Service
5 FederalDBLookupProcessor Customer identifiers Federal DB validation Federal DB Adapter
6 COTDeterminationProcessor Customer info, Federal DB results Class-of-trade Knowledge Base Service (DDD Matrix)
7 SAPCustomerCreationProcessor Customer details, COT Customer creation request SAP 4 HANA API
8 ChargebackResolutionProcessor Customer creation result Chargeback resolution, Compliance docs Model N API, Audit Trail

2.2.2 Conditional Logic

Workflow Routing:
  • If customer exists in SAP (Step 2): Skip to resolution (99.9% path)
  • If customer not eligible (Step 3): Route to exception queue (0.1% path)
  • If federal DB lookup fails: Retry with exponential backoff, then manual review
  • If COT determination ambiguous: Flag for human review with recommendations

2.2.3 Performance Requirements

  • Target Processing Time: <15 minutes per kickout (down from 25 minutes)
  • Throughput: Support 50-200 chargebacks per day
  • Success Rate: 99.9% automated resolution rate
  • Exception Rate: <0.1% requiring human review

Compliance:
  • Automatic capture of API responses for audit trail
  • Store screenshots/responses as document metadata
  • Maintain complete query history with timestamps
Error Handling:
  • Rate limit detection and automatic throttling
  • Timeout handling (30 second timeout per request)
  • Graceful degradation on API failures

2.4 Medicaid Forecasting Data Pipeline NEW

2.4.1 Data Sources

  • 867 Sellout Data: Weekly/monthly aggregated sellout data with shorter reporting lag
  • Claim-Level Dispense Data: Detailed Medicaid utilization data (3+ month lag)
  • Provider Master Data: Provider ID mappings and hierarchies

2.4.2 Provider ID Matching Service

Matching Logic:
  • Fuzzy matching on provider name, address, NPI
  • Hierarchical matching (state/region level if provider-level fails)
  • Confidence scoring for matches
  • Manual review queue for low-confidence matches

2.4.3 Lead/Lag Relationship Analysis

  • Analyze correlation between 867 sellout and Medicaid utilization
  • Determine optimal lead/lag periods by brand and geography
  • Model at multiple granularities (provider, state, national)

2.4.4 Forecasting Model Specifications

Specification Requirement
Forecast Target Medicaid utilization (units dispensed) per brand at national level
Brand Coverage All active Novartis brands with 2+ years utilization history
Prediction Horizon ~4 months ahead of reported utilization
Prediction Unit Week or month (determined during discovery)
Accuracy Target Improve GTN forecast accuracy to achieve 1% variance in out-of-period adjustment

2.5 Medicaid Rebate Invoice Processing ENHANCED

2.5.1 LLM Term Extraction

Extracted Fields:
  • Invoice date, Claimant/reporting entity
  • NDC codes, Units of measure (UOM)
  • ID codes (provider, pharmacy, etc.)
  • Units dispensed, Provider ID
  • Rebate amount requested

2.5.2 Error Detection Model

Error Type Detection Method Priority
UOM Errors Rule-based validation High
Duplicates within Invoice NDC + Date matching High
Duplicates across Invoices Historical invoice matching High
340B Duplicates Medicaid exclusion file check (Phase 1B) Medium

2.5.3 Triage Model Specifications

  • Focus: Sub-500K claims (80% of invoices, 11% of rebate dollars)
  • Output: Probabilistic error score and recommendation (approve/dispute)
  • Threshold: Configurable probability threshold for flagging (determined via backtesting)
  • Performance Target: Reduce review time from 60 minutes to <10 minutes per invoice
